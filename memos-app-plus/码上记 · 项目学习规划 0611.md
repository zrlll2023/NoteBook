**Phase 4 最需要注意的：**

1:文章列表的**动态查询**——用户可以按分类、标签、关键词任意组合筛选，条件是动态的，不是固定的。

比如：

- 只按分类查
- 只按关键词查
- 分类 + 关键词一起查

这里会用到 `LambdaQueryWrapper`，它可以根据条件是否存在动态拼接 SQL，是这个项目里 MyBatis-Plus 最核心的用法，面试也经常问。

2:文章新增的时候，需要同时往 `article` 和 `article_tag` 两张表写数据——这涉及到**事务**。

简单说：如果文章存进去了，但标签关联没存成功，数据就不完整了。用 `@Transactional` 注解可以保证「要么都成功，要么都回滚」。

### Phase 5 · Redis 集成

**5-1 热门文章列表缓存**

- 查询文章列表时，先查 Redis
- Redis 没有 → 查 MySQL → 结果存入 Redis，TTL 设置 10 分钟
- Redis 有 → 直接返回，不查 MySQL

**5-2 浏览量统计**

- 用户访问文章详情时，用 `INCR` 命令给 Redis 里的浏览量 +1
- 定时任务每隔一段时间把 Redis 里的浏览量同步回 MySQL

**5-3 黑名单完善**

- 验证登出和拦截器黑名单的完整闭环（Phase 3 已做，这里做联调测试）

**Phase 5 最需要注意的：**

1:缓存和数据库的**数据一致性**——

Redis 里缓存了文章列表，但如果有人新发了一篇文章，Redis 里的数据就过时了。

怎么处理？两个方案：

- TTL 到期自动失效（我们用这个，简单）
- 发文章时主动删除 Redis 缓存

理解这个取舍，面试时能说清楚很重要。

2:浏览量用 Redis `INCR` 而不是直接更新 MySQL，原因是：

- MySQL 每次 `UPDATE` 都要加行锁
- 高并发下 1000 个人同时看同一篇文章，1000 次加锁会很慢
- Redis 是单线程的，`INCR` 是原子操作，天然解决并发问题

面试如果问「为什么用 Redis 做浏览量统计」，这就是标准答案。

**Phase 6 最需要注意的：**

1:`docker-compose.yml` 里三个服务的**启动顺序和网络配置**——

App 依赖 MySQL 和 Redis，所以 MySQL 和 Redis 必须先启动。如果配置不对，App 启动时找不到数据库会报错。

2:Dockerfile 里有一个**多阶段构建**的概念——

第一阶段用 Maven 镜像编译打包，第二阶段只用 JRE 运行 jar 包，最终镜像体积会小很多。

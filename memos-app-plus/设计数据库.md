库名：memosplus_db
字符集：utf8mb4
排序规则：utf8mb4_general_ci

表：
#### users 用户表(`user` 是 MySQL 的保留关键字，直接用会报错！)
- `id` 主键
- `username` 用户名
- `password` 密码
- `avatar` 头像
- `status` 状态（0 = 正常，1 = 禁用）
- `created_time` 注册时间
- `updated_time` 更新时间
#### article 文章表
- `id` 主键
- `title` 文章标题
- `content` 文章内容
- `user_id` 文章对应的用户id
- `category_id` 分类对应的id
- `view_count` 浏览量
- `created_time` 创建时间
- `updated_time` 更新时间
#### category 分类表
- `id` 主键
- `name` 分类类型
#### tag 标签表
- `id` 主键
- `name` 标签内容
#### article_tag 文章标签关联表
- `id` 主键
- `article_id` 对应文章的id
- `tag_id` 对应标签的id
#### comment 评论表
- `id` 主键
- `article_id` 对应文章的id
- `comment_content` 评论内容
- `user_id` 评论的用户对应的id
- `comment_time` 用户评论的时间



---
`INT` 类型最大能存的数字是 **2,147,483,647**，大概是 21 亿。

听起来很多对吧？但是想一下——

> 如果你的平台做大了，用户量达到几十亿（比如微信），INT 存满了怎么办？

而 `BIGINT` 最大能存 **9,223,372,036,854,775,807**，大概是 922 亿亿，基本上不可能用完。

---

所以用 `BIGINT` 做主键是一个**防患于未然**的习惯——现在多占一点空间，以后不用担心 ID 溢出的问题。

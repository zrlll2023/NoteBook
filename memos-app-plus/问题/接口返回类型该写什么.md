这问题问得对,值得停下来讲清楚——因为「接口该返回什么
  类型」不是背的,是推出来的。用直接讲解,这是个你目前没
  料可被引导的点。

  先看一个你已经做对的事实——你自己写的查询方法:

  public Result<List<Category>> getAll() {
      ...
      return Result.success(categories);
  }

  你这里返回类型写的是
  Result<List<Category>>。为什么是它?
  倒推一下:你方法体里 return
  Result.success(categories),返回的是一个 Result
  对象,里面装着 List<Category>。所以返回类型 = 你
  return 出去的那个东西的类型。这就是规则,没有玄学:

  【知识点定位】 Java > 方法签名 > 返回类型与 return
  语句的类型一致性
  【标准定义】
  方法声明的返回类型,必须能接得住方法体里每一条 return
  语句返回的值。返回类型是「我承诺给调用方什么」,retu
  rn 是「我实际给了什么」,两者必须对得上。

  ---
  现在回到你的新增方法。先回答一个更上层的问题:这个项
  目里,所有接口对外统一返回什么?

  去翻你 UserController——register、login、logout
  三个方法,返回类型分别是什么?你会发现一个规律:这个项
  目里每个接口都返回
  Result<...>,没有一个直接返回裸对象。这是你
  README「项目亮点」里写的统一返回格式——前端永远收到
  {code, message, data}
  这个结构,好处是前端处理逻辑统一。

  所以新增方法返回 Category(裸对象)从一开始就违反了这
  个项目的约定,第一步先确定:它必须返回
  Result<某个泛型>。

  第二步,定那个泛型 <> 里填什么。 这取决于一件事:

  ▎ 新增一个分类成功后,前端需不需要拿到数据?需要什么数
  ▎ 据?

  你看你方法体里写的是 Result.success(null)——你没打算
  给前端返回任何数据(传了 null)。那你对照一下
  UserController 里同样「不返回数据」的那个方法(logout
  ),它的返回类型泛型填的是什么?

  你来答这一个问题就行:既然 success(null)
  不带数据,泛型 <>
  里该填哪个类型?填出来,你新增方法的返回类型就完整了。
